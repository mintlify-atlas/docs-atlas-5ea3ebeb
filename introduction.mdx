**Broccoli is a robust message queue system for Rust applications**, designed as a modern alternative to Celery with full async/await support, type safety, and multiple broker backends. It's built for developers who want Rust's performance and type guarantees in their job processing layer.

## Why Broccoli?

If you're familiar with Celery but want to leverage Rust's performance and type safety, Broccoli is your answer. While Celery is a powerful, battle-tested job queue for Python, it has limitations in compiled languages:

- **Weak typing**: Python's dynamic typing can lead to runtime errors in Celery tasks
- **GIL constraints**: Python's Global Interpreter Lock limits concurrency
- **Performance overhead**: Serialization and network overhead in distributed Python systems
- **Language barrier**: Celery doesn't integrate natively with Rust's ownership model

Broccoli brings the familiar Celery paradigm to Rust, with a consistent API across multiple message brokers.

## Broccoli vs. Celery

| Aspect | Broccoli | Celery |
|--------|----------|--------|
| **Language** | Rust (compiled) | Python (interpreted) |
| **Type Safety** | Full compile-time typing | Runtime validation |
| **Performance** | High (native binaries) | Moderate (GIL overhead) |
| **Async/Await** | Native throughout | Thread/gevent-based |
| **Brokers** | Redis, RabbitMQ, SurrealDB | Redis, RabbitMQ, many others |
| **Learning Curve** | Moderate (Rust syntax) | Gentle (Python simplicity) |
| **Maturity** | Active development | Battle-tested (10+ years) |

**Choose Broccoli if you:**
- Want Rust's safety and performance in a job queue
- Prefer compiled, type-safe task definitions
- Need native async/await throughout
- Are building Rust services and want consistency

**Celery remains better if you:**
- Prefer Python's simplicity and development speed
- Need broader ecosystem integration
- Rely on Celery-specific plugins or third-party extensions

## Architecture Overview

Broccoli follows a classic producer-consumer pattern:

1. **Publishers** send serialized messages to a broker
2. **Brokers** persist messages (Redis, RabbitMQ, or SurrealDB)
3. **Consumers** fetch and process messages with configurable concurrency
4. **Error handling** supports retries, custom handlers, and failure recovery

All communication is async-first, allowing high throughput with minimal resource consumption.
